너는 이 저장소의 `docs/` 문서를 **실제 코드 변경에 맞춰** 정확히 갱신하는 작업을 한다.

## 입력
- 사용자 인자(선택): `$ARGUMENTS`
  - 비어 있으면 아래 git 변경 내역에서 스스로 범위를 판단한다.
  - 도메인명(예: `ticket`)이나 변경 요약이 주어지면 그 범위를 우선한다.

## 현재 변경 컨텍스트 (자동 수집)
- 작업 트리 상태:
!`git status --short`
- 스테이지/언스테이지 변경 요약:
!`git diff --stat HEAD`
- 최근 커밋:
!`git log --oneline -10`

## 문서 규칙 (반드시 준수)
- 문서 인덱스/추가 규칙: @docs/README.md
- 번호 체계와 "공통 기반 / 도메인별" 구분, 신규 도메인 문서 추가 규칙(`NN-{domain}-*.md`)을 그대로 따른다.

## 진행 절차
1. **범위 파악**: 위 git 변경과 `$ARGUMENTS` 를 종합해 어떤 코드가 바뀌었는지 식별한다.
   필요하면 Glob/Grep/Read 로 실제 소스(`src/main/kotlin/**`, `build.gradle.kts`, `*.yml`, `compose.yaml`)를 확인한다.
2. **영향 문서 결정**:
   - 기존 기능 수정/확장 → 해당 도메인의 기존 문서를 **Edit** 로 갱신.
   - 새 도메인 추가 → README 의 "문서 추가 규칙"에 따라 **다음 번호부터** 새 `.md` 들을 **Write** 로 생성.
   - 공통 기반(01~03, 10)에 해당하는 변경(버전/빌드/아키텍처/설정)도 누락 없이 반영.
3. **인덱스 동기화**: `docs/README.md` 의 표/섹션과 "✅ 현재까지 구현된 기능", "🗺️ 앞으로" 항목을 최신화한다.
4. **정확성 검증**: 코드에 없는 내용을 추측해서 쓰지 않는다. 클래스명/경로/엔드포인트/상태코드는 실제 소스와 대조해 일치시킨다.

## 출력 형식
- 작성/수정한 문서 파일 목록과 각 파일의 핵심 변경점을 한국어로 간단히 요약해 보고한다.
- 코드와 문서가 어긋나는 부분을 발견하면(예: 문서엔 있는데 코드에서 사라진 API) 별도로 지적한다.

## 주의
- 문서 파일(`docs/**`, `.claude/**`)과 직접 관련된 것 외의 **소스 코드는 수정하지 않는다.**
- 한국어로 작성하고, 기존 문서의 표기/스타일(표·코드펜스·이모지 헤더)을 따른다.
